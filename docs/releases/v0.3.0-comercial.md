# Release Review v0.3.0 Comercial

Auditoria realizada na branch `feature/v030-formalizacao-comercial`.

Esta revisao nao altera codigo, telas, CSS, Firebase ou regras de negocio. O
objetivo e registrar o estado da v0.3.0 antes do inicio da v0.4 Producao.

## 1. Resumo da versao

A v0.3.0 consolida o bloco de Formalizacao Comercial do RK-Conecte. A versao
parte da base comercial criada na v0.2.x e completa o caminho do Orcamento
Inteligente ate a aprovacao, conversao em Projeto Executivo e leitura por
Dashboard Comercial.

O fluxo novo esta arquiteturalmente coerente: usa models, validators, services,
use cases, controllers e renderers separados. Os modulos recentes respeitam o
limite de nao acessar Firestore diretamente e mantem o Orcamento Inteligente
como fonte principal dos dados comerciais.

## 2. Objetivo da v0.3.0

Preparar uma proposta comercial profissional e rastreavel, conectando:

- dados comerciais de Cliente, Projeto, Servico e Produto;
- calculo e resumo do Orcamento Inteligente;
- Documento Comercial estruturado;
- renderizacao visual;
- PDF Comercial;
- aprovacao comercial;
- conversao em Projeto Executivo;
- leitura gerencial por Dashboard Comercial.

## 3. Principais entregas

- App State Manager em `js/core/`.
- Document Pipeline em `js/documentos/`.
- Document Renderer e HTML Renderer.
- Export Service e adapters independentes.
- PdfAdapter com `pdf-lib` encapsulado.
- Central de Compartilhamento do Documento Comercial.
- Aprovacao Comercial com estados proprios.
- Conversao em Projeto Executivo.
- Dashboard Comercial somente leitura.
- Documentacao atualizada em changelog, Project Bible e READMEs dos modulos.

## 4. Fluxo comercial implementado

```text
Cliente
  -> Projeto
  -> Servico
  -> Produto
  -> Orcamento
  -> Documento
  -> PDF
  -> Aprovacao
  -> Conversao em Projeto
  -> Dashboard
```

Leitura por modulos:

- Cliente: dominio, repository, service, use cases e tela comercial.
- Projeto: modelo central, service e storage de transicao.
- Servico: dominio, repository, service, use cases e catalogo.
- Produto: dominio, repository, service e use cases.
- Orcamento: Orcamento Inteligente orquestra Cliente, Projeto, Servico,
  Produto, Motor de Calculo e Workflow.
- Documento: Document Pipeline normaliza o Documento Comercial.
- PDF: Export Service seleciona adapter; PdfAdapter encapsula `pdf-lib`.
- Aprovacao: ComercialService controla `RASCUNHO`, `EM_REVISAO`, `APROVADO` e
  `REPROVADO`.
- Conversao: ConversaoService cria Projeto Executivo via ProjetoService.
- Dashboard: Dashboard Comercial consulta AppState e services existentes.

## 5. Arquitetura atual

### Core

- `EventBus` oferece `on`, `off`, `emit`, `once` e `clear`.
- `AppState` centraliza estado em memoria e emite eventos quando EventBus esta
  disponivel.
- `AppStateService` funciona como fachada de acesso.
- `EventTypes` ainda lista apenas parte dos eventos usados pelos modulos novos.

### Workflow

- Workflow segue isolado em `js/workflow/`.
- `WorkflowEngine` valida transicoes do ciclo de vida do Projeto e registra
  eventos em memoria.
- Aprovacao Comercial usa estados proprios, sem alterar a maquina de Projeto.

### Repository e Storage

- Cliente, Servico e Produto seguem Repository + Adapter.
- `MemoryAdapter` esta funcional.
- `FirestoreAdapter` permanece reservado.
- `ProjetoService` ainda usa `ProjetoStorage`, que mistura LocalStorage e
  Firestore como ponte de compatibilidade.

### Dominios comerciais

- Cliente, Servico e Produto estao bem separados em model, validator,
  repository, service e use cases.
- Projeto e a entidade central, mas ainda nao esta totalmente alinhado ao mesmo
  padrao Repository + Adapter dos outros dominios.

### Formalizacao Comercial

- Document Pipeline, Renderer, Export Service, ComercialService,
  ConversaoService e Dashboard Comercial seguem arquitetura modular.
- AppState guarda dados auxiliares em `configuracoes`, preservando as chaves
  originais do estado.
- Conversao gera Projeto Executivo em memoria via `ProjetoService.criarManual`.

## 6. Pontos fortes

- Fluxo comercial de ponta a ponta existe e esta documentado.
- Modulos novos evitam Firestore direto.
- `pdf-lib` esta encapsulado apenas no PdfAdapter.
- Document Pipeline criou um contrato comum para HTML, PDF, impressao e
  compartilhamento.
- Aprovacao e Conversao usam Documento Comercial como fonte de dados.
- Dashboard Comercial e somente leitura e nao altera AppState.
- EventBus e Workflow ja permitem rastreabilidade minima.
- Controllers e UIs novas foram mantidos separados.
- A documentacao acompanhou a evolucao sprint a sprint.

## 7. Pontos de atencao

- A aplicacao ainda depende da ordem manual de `<script>` nas paginas.
- `ProjetoStorage` ainda acessa Firestore diretamente dentro da camada de
  storage, mesmo que as telas novas nao acessem Firestore.
- Scripts legados ainda usam `db.collection(...)` diretamente.
- WorkflowEvents e EventBus continuam separados.
- `ProjetoStatus` e `WorkflowState` mantem conceitos parecidos.
- AppState e usado como memoria de integracao, mas nao persiste dados.
- A Conversao em Projeto ainda nao salva o Projeto em repository/storage.
- Dashboard Comercial consulta dados disponiveis, mas nao possui fonte historica
  consolidada.
- Central de Compartilhamento ainda tem WhatsApp, Email e Link como
  placeholders.

## 8. Dividas tecnicas conhecidas

- Migrar Projeto para Repository + Adapter.
- Implementar FirestoreAdapter real ou definir oficialmente ProjetoStorage como
  camada de compatibilidade temporaria.
- Criar testes automatizados para models, validators, services, adapters e
  fluxos compostos.
- Criar uma estrategia oficial para build/publicacao de dependencias de
  navegador.
- Consolidar eventos entre WorkflowEvents e EventBus.
- Padronizar eventos novos em `EventTypes`.
- Separar ou modularizar CSS em sprint propria.
- Corrigir encoding inconsistente em documentos e telas legadas.
- Migrar acessos diretos a Firestore em scripts antigos.
- Formalizar persistencia definitiva de Documento Comercial, Aprovacao e
  Conversao.

## 9. Riscos antes da v0.4

- Risco alto: `pdf-lib` esta em `package.json`, mas o Firebase Hosting ignora
  `node_modules`. Como o PdfAdapter tenta carregar o UMD a partir de
  `node_modules` no navegador quando `PDFLib` nao existe globalmente, a geracao
  real de PDF pode falhar no ambiente publicado.
- Risco medio: Conversao em Projeto cria estrutura executiva em memoria, mas a
  v0.4 Producao precisara de persistencia confiavel para consumir esse Projeto.
- Risco medio: ausencia de testes automatizados aumenta chance de regressao em
  fluxos longos, especialmente Orcamento -> Documento -> PDF -> Aprovacao ->
  Conversao.
- Risco medio: scripts legados com Firestore direto podem continuar gerando
  efeitos colaterais se forem usados junto com o fluxo novo.
- Risco baixo/medio: Dashboard Comercial ainda mede apenas dados disponiveis em
  AppState e ProjetoService, nao uma base historica completa.

## 11. Recomendacao final

**A v0.3.0 esta pronta para ser tagueada como `v0.3.0-comercial`?**

**NAO, ainda nao como release pronta.**

Justificativa: a arquitetura comercial esta madura e o fluxo funcional esta
bem encadeado, mas ha um risco concreto na entrega do PDF real em ambiente
publicado porque `node_modules` e ignorado pelo Firebase Hosting enquanto o
PdfAdapter depende desse caminho como fallback no navegador. Alem disso, a
Conversao em Projeto ainda nao possui persistencia definitiva e nao ha suite
automatizada para proteger o fluxo completo antes da entrada em Producao.

Recomendacao pratica: criar a tag somente depois de decidir a estrategia de
publicacao do `pdf-lib` para browser e registrar claramente se a conversao em
Projeto da v0.3.0 e um marco em memoria ou uma entrega persistida.
